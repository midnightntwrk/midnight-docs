# Community wallets

Community-built wallet integrations for interacting with the Midnight Network. These projects extend wallet support beyond the official Midnight Lace wallet.

:::warning[Community-maintained - not officially supported]

Their respective authors maintain the projects listed here, not the Midnight Foundation. Verify compatibility against the [compatibility matrix](/relnotes/support-matrix) before integrating. File issues with each project's maintainers directly.

:::

## Choosing and integrating a wallet

A reference for choosing and integrating a wallet on the Midnight Network for a DApp, a backend, an
agent, or personal use. Instead of just listing wallets, it gives you a **mental model** for the whole
landscape (a **custody x interface matrix**), shows which cells each wallet covers (its *coverage
footprint*), and lets you pick by *scenario*, then keeps the per-wallet detail and integration code as
reference.

Midnight is a Cardano partnerchain built by IOG: a dual ledger (public *unshielded* + private
*shielded* state), two tokens (**NIGHT** for value, **DUST** for fees), and **Compact** smart contracts
that compile to zero-knowledge circuits. This guide assumes you know those basics. Prefer video? See the
[Lace & 1AM walkthrough](https://www.youtube.com/watch?v=m7mU7k5lgqU) (Edda Labs).

|---|---|
| **Guide version** | v1 — 2026-06 |
| **DApp Connector API** | `@midnight-ntwrk/dapp-connector-api` **v4.0.1** |
| **Local proof server** | `http://localhost:6300` (Lace); CLI connector on `ws://localhost:9932` |
| **Networks** | `mainnet`, `preview`, `preprod`, `undeployed` (local) — only `'mainnet'` is connector-standard; other ids are wallet-defined ([§5.2](#52-connect-and-the-connectedapi-surface)) |

:::tip[Start here]

In practice, today's decision collapses to a **single custody row, self-custody,**
because the other three models (custodial, smart-contract/AA, MPC) are barely populated on Midnight yet;
you only choose *how to reach it*: a **browser**, the **CLI**, or an **AI agent**. So: for a
**browser dApp**, integrate **1AM + Lace** via the DApp Connector. For **backend / CI / agents**, use
the **wallet CLI** (`midnight serve` + its MCP server). For **institutional** custody of NIGHT,
**Fireblocks**. Everything else is a variant of those.

- **Privacy by default?** **1AM** (shielded-by-default, in-browser proving).
- **Mobile-first / embedded wallet?** **Kuira** (Android SDK, on-device proving — [§6](#kuira)).
- **One multi-chain wallet?** **Ctrl** (feature-detect its connector).
- **Consumer wallet of the future?** **Midnight Passport** (announced — [§6](#midnight-passport-announced)).
- **urble** is a consumer privacy-savings app, not a dApp-integration target.

:::

The Midnight wallet ecosystem evolves quickly — some integrations are newly launched and
others are **announced** or **not yet publicly documented**, and this guide labels them as such. Details
are accurate as of **June 2026**; for the latest, follow the linked [Sources](#sources) and the
[Midnight forum](https://forum.midnight.network/).

## 1. How to read this guide

Ask **two questions** about any wallet and you've placed it on the map:

1. **Custody: who controls the keys?** (the *trust* axis) On the user's device, held by a provider,
   governed by a contract, or split into shares?
2. **Interface: how is it reached?** (the *access* axis) A human in a **browser** or **mobile** app, an
   automation via **CLI/SDK**, or an **AI agent** via **MCP**?

Put those on a grid → a **4 × 4 matrix** ([§2](#2-the-mental-model-custody-and-interface)). Every wallet
fills **one or more cells** — its *coverage footprint* ([§3](#3-reading-a-wallet-as-a-coverage-footprint)).
Choosing then becomes "find the cell my scenario needs, pick a wallet that covers it"
([§4](#4-choose-by-scenario)). The rest is reference: how to integrate ([§5](#5-integrate)), the wallets
in detail ([§6](#6-the-wallets)), the feature matrix ([§7](#7-functionality-matrix)), and the dev
workflow / troubleshooting / security that follow.

A secondary lens for integration — the seven detailed wallets fall into three product categories:

| Category | Wallets | What it means for you |
|----------|---------|------------------------|
| **Midnight-native** | 1AM, urble, Kuira | Built for Midnight's dual-state / ZK model; shielded-capable. 1AM has the browser dev-integration story; urble is an end-user app; **Kuira** is mobile (Android) + an embeddable SDK ([§6](#kuira)). |
| **Cardano multi-chain + Midnight** | Lace, Ctrl, Gero | Established wallets bolting on Midnight. Lace is the IOG reference; Ctrl/Gero are third-party. |
| **Embedded / Wallet-as-a-Service (WaaS)** | Dynamic, Kuira | An SDK you embed to mint wallets. Dynamic is social/passkey **WaaS** (Wallet-as-a-Service) with Midnight support **not first-party documented** ([§6](#dynamic)); **Kuira** is a Midnight-native, self-custody Android SDK ([§6](#kuira)). |

Building a **browser** dApp? The connectors you can target today are **Lace** and **1AM** (Ctrl connects
per its own claims; Gero/others appear in the ecosystem catalog).

## 2. The mental model: custody and interface

### 2.1 Why two axes

- **Custody (rows) = the trust model.** *Where do the keys live, and who can sign?* This decides what can
  go wrong and who is in control. It maps onto Midnight's custody spectrum —
  **Full Sovereignty → Delegated → Smart Custody**.
- **Interface (columns) = how the wallet is reached.** The first two are how **humans** interact
  (browser, mobile); the third is how we **automate** (CLI/SDK); the fourth is how **AI agents** act
  (MCP). Same keys, different access patterns and threat models.

The axes are independent — a self-custody key can be reached from a browser, a CLI, or an agent — which
is why a grid fits, and why one wallet often spans several cells.

The four custody rows line up with Midnight's three-point custody spectrum like this:

| Matrix custody row | Midnight spectrum | Example on Midnight |
|---|---|---|
| Self-custody | Full Sovereignty | Lace, 1AM, Kuira |
| Custodial | Delegated | (none for consumers yet) |
| Smart contract / AA | n/a (not native) | n/a |
| MPC / threshold | Smart Custody | Fireblocks (NIGHT) |

### 2.2 The four custody models

**1. Self-custody: keys on the user's device.** Key lives in the extension, mobile secure storage, or
hardware. No third party can sign. *Trust yourself.* Full control and privacy; lose the seed phrase, lose
the funds. **On Midnight:** where almost every wallet lives today (Lace, 1AM, Ctrl, Gero, SubWallet,
NuFi). Maps to **Full Sovereignty**.

**2. Custodial — a provider holds the keys.** An exchange/fintech signs on the user's behalf. *Trust the
custodian* (single point of failure); easy onboarding, counterparty risk, no privacy from the provider.
**On Midnight:** no first-party *consumer* custodial wallet documented yet; the custodial path today is
institutional (MPC, below) + exchange/Wallet-IaaS (managed wallet provisioning — the same idea as WaaS) as
the network matures. *(An emerging area on Midnight.)*

**3. Smart-contract / Account Abstraction (AA) — the account is a contract.** Programmable authorization
(multisig, social recovery, session keys, sponsored fees). *Trust the contract code.* **On Midnight:**
**no native ERC-4337-style AA today** — Midnight's programmability lives in **Compact contracts** + its
dual-state privacy model, which is *orthogonal* to account-level abstraction. So this row is **Not yet on
Midnight** *(illustrative: Safe, ERC-4337)*. Some AA-*like* UX can still be approximated, though. Sponsored
fees come via the connector's relayer, and social login via embedded wallets (Wallet-IaaS / [Dynamic](#dynamic)).
An **embedded-SDK** approach — where the app itself carries a self-custody wallet ([Kuira](#kuira)) — gives a
similar "no separate wallet, no seed" feel. But **none of these is a smart-contract account**, so the row
stays *Not yet*.

**4. MPC / threshold — the key is split into shares.** No single share signs; an *m*-of-*n* quorum
produces the signature without reconstructing a full key. *Trust distributed across shareholders;*
institutional-grade governance. **On Midnight:** the **institutional** path — **Fireblocks** (MPC-based)
announced custody of **NIGHT**, letting clients securely claim/hold it (e.g. during the Glacier Drop —
Midnight's NIGHT token distribution event).
([Midnight × Fireblocks](https://midnight.network/blog/midnight-foundation-collaborates-with-fireblocks-to-expand-institutional-access-to-web3),
[Fireblocks MPC 101](https://www.fireblocks.com/report/what-is-mpc)) Embedded SDKs like Dynamic also use
**TSS** (threshold signature scheme) MPC. Maps to **Smart Custody**.

### 2.3 The four interfaces

- **Browser** — extension / web UI, human and interactive; where the **DApp Connector**
  (`window.midnight`) lives ([§5](#5-integrate)). *On Midnight:* Lace, 1AM, Ctrl, Gero, SubWallet, NuFi.
- **Mobile native** — biometric, secure-enclave keys, on the go. *On Midnight:* 1AM (beta), urble, and
  **Kuira** (Android, alpha — on-device proving + an embeddable SDK; [§6](#kuira));
  **Lace mobile is Cardano/Bitcoin only — Midnight is the desktop extension.**
- **CLI** — scriptable, unattended automation: backend payouts, CI, and **building/testing with no
  extension** (the wallet CLI exposes the same `ConnectedAPI` over a local WebSocket —
  [§8](#8-develop-and-test-without-a-wallet)). *Note — two senses of "SDK":* this column is the
  **automation** SDK/CLI (the wallet CLI). An **app-embed** SDK (e.g. [Kuira](#kuira)'s Android library)
  is a delivery mechanism for the **Mobile** cell — a wallet built *into* an app — not a separate column.
- **MCP** — an AI agent calls **tools**, in two flavors: *wallet-control* (an agent that holds/moves
  funds — the wallet CLI's MCP server) and *dev-assistant* (an agent that helps you build —
  [§9](#9-ai-and-agent-tooling-the-mcp-column)). The **newest** interface — and the cell that most needs guardrails.

### 2.4 The matrix

Legend: **Available** (a real, usable Midnight product — incl. beta/alpha) · **Emerging** (announced or
institutional — not yet a shippable self-custody product) ·
**Not yet** (none on Midnight — an *italic* industry example illustrates) · **—** (n/a).

| Custody ↓ / Interface → | Browser | Mobile Native | CLI (SDK / CI) | MCP (AI agents) |
|---|---|---|---|---|
| **Self-custody** (keys on device) | **Available** — Lace, 1AM, Ctrl, Gero, SubWallet, NuFi | **Available** — 1AM (beta), urble, **Kuira** (Android, alpha) | **Available** — midnight-wallet-cli (`midnight serve`) | **Available** — midnight-wallet-cli MCP server |
| **Custodial** (provider holds keys) | **Not yet** · *CEX web wallets* | **Not yet** · *CEX apps* | **Not yet** · *exchange custody APIs* | **Not yet** · *provider agent tooling* |
| **Smart-contract / AA** (account is a contract) | **Not yet** · *Safe, ERC-4337* | **Not yet** | **Not yet** | **Not yet** |
| **MPC / threshold** (key split into shares) | **Emerging** — *Midnight Passport* (announced; seedless — [§6](#midnight-passport-announced)); institutional console | **Emerging** — *Midnight Passport* (announced) | **Emerging (institutional)** — Fireblocks; Dynamic (TSS-MPC; Midnight *not documented*) | **Emerging** — MPC + agent custody *(industry)* |

**Reading the grid:** the **entire self-custody row is live today** — reachable from browser, mobile, CLI,
and AI agents; most dApps target **Self-custody × Browser** (and Mobile). The **AA
row is empty** (no native smart accounts). The **MPC row is mostly institutional** (Fireblocks), with one
*consumer* entry on the way — **Midnight Passport** (its full technical details are not yet public;
[§6](#midnight-passport-announced)). **Empty cells are a roadmap**, not a dead end.

:::note[The ecosystem evolves quickly]

For the current state of wallet and custody support, see the live
[ecosystem catalog](https://midnight.network/ecosystem-catalog) (use the
[institutional-custody](https://midnight.network/ecosystem-catalog?tag=institutional-custody) tag for
the custody/MPC slice).

:::

## 3. Reading a wallet as a coverage footprint

**A wallet is a set of cells, not a point.** Scan the cells it fills and you see *who can use it and how*;
the empty cells tell you what it won't do. Three contrasting footprints:

**1AM — a person's wallet** (human interfaces, private by default)

| Custody \ Interface | Browser | Mobile | CLI | MCP |
|---|---|---|---|---|
| **Self-custody** | ✓ (Chrome/Firefox) | ✓ beta (iOS/Android) | — | — |

→ "a consumer + dApp wallet for desktop and phone, shielded by default." (Drivable from code via the
connector, but no first-party CLI/MCP.)

**midnight-wallet-cli — the automation & agent footprint**

| Custody \ Interface | Browser | Mobile | CLI | MCP |
|---|---|---|---|---|
| **Self-custody** | — | — | ✓ commands + `midnight serve` | ✓ 31-tool MCP server (contract deploy/call + spend confirmation) |

→ "not a consumer wallet — it's how scripts, CI, and agents hold and move funds." One network-agnostic
seed serves all networks. Reach for it in [§8](#8-develop-and-test-without-a-wallet) and
[§9](#9-ai-and-agent-tooling-the-mcp-column).

**Lace — a focused reference footprint**

| Custody \ Interface | Browser | Mobile | CLI | MCP |
|---|---|---|---|---|
| **Self-custody** | ✓ (Chrome/Edge) | — *(Cardano only)* | — | — |

→ "the safe desktop default; its mobile/hardware reach is Cardano-side — pair it with the wallet CLI for
automation." Narrow isn't bad: it tells you exactly where Lace fits.

**Kuira — a mobile footprint shipped as an embeddable SDK** (the wallet apps carry inside themselves)

| Custody \ Interface | Browser | Mobile | CLI | MCP |
|---|---|---|---|---|
| **Self-custody** | — | ✓ Android (alpha) | — | — |

→ "a wallet that ships *inside* an Android app — on-device proving, passkey identity, no separate
install." A genuinely different shape: not a tile you connect to, but an SDK you build *into* a mobile
app. (Its embeddable SDK is *how the Mobile cell ships*, not a CLI/automation surface — see the note in
[§2.3](#23-the-four-interfaces).)

:::tip[Takeaway]

Sketch the footprint before comparing features — *where* a wallet sits and *how far* it
reaches answers most of "is this right for my scenario?". The matrix in
[§7](#7-functionality-matrix) is for the tie-breakers.

:::

## 4. Choose by scenario

Answer two questions, land on a cell, pick a wallet whose footprint covers it.

**Who controls the keys?** end user → **Self-custody** · provider → **Custodial** *(not on Midnight for
consumers)* · institution/DAO → **MPC** · programmable account → **AA** *(not native yet)*.
**Who reaches it?** person in a browser → **Browser** · on a phone → **Mobile** · script/CI → **CLI** ·
AI agent → **MCP**.

The same two questions as a quick decision tree:

```
Who controls the keys?
  ├─ end user (self-custody) ──► How is it reached?
  │                                browser    → 1AM + Lace
  │                                mobile     → 1AM / Kuira (Android)
  │                                script/CI  → wallet-CLI (midnight serve)
  │                                AI agent   → wallet-CLI MCP server
  ├─ institution / DAO ─────────► MPC → Fireblocks
  └─ programmable account ──────► AA  → not native yet
```

| Your scenario | Cell(s) | Reach for |
|---|---|---|
| Consumer privacy dApp; desktop & phone | Self-custody × Browser/Mobile | **1AM** (+ **Lace**); **Kuira** for Android; **Ctrl** for multi-chain |
| Mobile-first app with a **built-in** wallet (no separate install) | Self-custody × Mobile (SDK) | **Kuira** (Android; iOS/RN later — [§6](#kuira)) |
| "One wallet for all my chains" | Self-custody × Browser/Mobile | **Ctrl** (feature-detect first) |
| Just claim / hold **NIGHT** (Cardano side) | Self-custody × Browser/Mobile | **Lace, Gero, Yoroi, Eternl, NuFi, Vespr…** ([§12](#12-broader-ecosystem-compatibility)) |
| Consumer privacy **savings** app | Self-custody × Mobile | **urble** |
| Backend payouts, airdrops, CI / E2E | Self-custody × CLI | **wallet CLI** / `midnight serve` ([§8](#8-develop-and-test-without-a-wallet)) |
| An **AI agent** that holds/moves funds | Self-custody × MCP | **wallet-cli MCP server** (with guardrails — [§9](#9-ai-and-agent-tooling-the-mcp-column)) |
| **AI-assisted building** | dev-assistant MCP | **Midnight MCP / Midnight Expert + Edda** ([§9](#9-ai-and-agent-tooling-the-mcp-column)) |
| Institutional / regulated custody | MPC × console / API | **Fireblocks** |
| Embedded social-/passkey onboarding | MPC / Smart-Custody | **Dynamic** *(confirm Midnight support)* or Wallet-IaaS |

Each wallet's **"Best for"** and feature breakdown is in [§6](#6-the-wallets); the feature matrix is
[§7](#7-functionality-matrix). Rules of thumb:

- **Browser dApp today?** Target **1AM and Lace** via the standard connector; add Ctrl/others only after
  runtime feature-detection.
- **Automating / agent-driving?** Use the **CLI / MCP** column (the wallet CLI), not a consumer extension.
- **Privacy by default?** 1AM and urble default to shielded; Lace and the multi-chain wallets are opt-in.
- **Who proves?** 1AM proves in-browser (`getProvingProvider`); **Kuira proves on-device** (Android);
  Lace needs a local proof server and does *not* expose it — feature-detect to support both
  ([§5.3](#53-where-zk-proofs-come-from)).

## 5. Integrate

Everything below is the **Self-custody × Browser** cell in depth — the connector model plus the code to
ship it. (For no-extension, headless, or agent flows, see [§8](#8-develop-and-test-without-a-wallet) /
[§9](#9-ai-and-agent-tooling-the-mcp-column).)

### 5.1 The DApp Connector API (CAIP-372)

A Midnight browser wallet injects an **Initial API** under the global `window.midnight`, keyed by an
identifier — compatible with the draft
[CAIP-372](https://github.com/ChainAgnostic/CAIPs/pull/372/files) (Chain Agnostic Improvement Proposal)
standard, so the *same dApp code* works across wallets.
([npm](https://www.npmjs.com/package/@midnight-ntwrk/dapp-connector-api),
[connector repo + SPEC](https://github.com/midnightntwrk/midnight-dapp-connector-api),
[API reference](https://docs.midnight.network/api-reference/dapp-connector))

```ts
type InitialAPI = {
  rdns: string;        // reverse-DNS id, stable per product
  name: string;        // display name (sanitize before rendering — XSS)
  icon: string;        // URL or data: URL (render via <img>, not innerHTML)
  apiVersion: string;  // version of @midnight-ntwrk/dapp-connector-api implemented
  connect: (networkId: string) => Promise<ConnectedAPI>;
};
```

**Discovery has two generations — support both.** Legacy direct key (Lace at `window.midnight.mnLace`,
1AM at `window.midnight['1am']`) **and** the v4 rdns scan (iterate keys, match on `name`/`rdns`,
EIP-6963-style — the Ethereum Improvement Proposal for multi-wallet discovery). ([React wallet-connect guide](https://docs.midnight.network/guides/react-wallet-connect))

### 5.2 Connect and the ConnectedAPI surface

`connect(networkId)` prompts the user and resolves to a **ConnectedAPI** spanning Midnight's three asset
kinds (*shielded*, *unshielded*, *DUST*). **Only `'mainnet'` is standardized by the connector spec** —
every other network id is **wallet-defined**, and they really do differ (the CLI connector, for example,
expects capitalized `'Preview'`/`'PreProd'`/`'Undeployed'` — which is why the [§8](#8-develop-and-test-without-a-wallet)
snippet uses `networkId: "Preview"`). Check your target wallet's docs for the exact strings.

| Method | Purpose |
|--------|---------|
| `getShieldedAddresses()` / `getUnshieldedAddress()` / `getDustAddress()` | Bech32m addresses per asset kind |
| `getShieldedBalances()` / `getUnshieldedBalances()` | per-token balances |
| `getDustBalance()` | `{ balance, cap }` — DUST regenerates toward `cap` |
| `makeTransfer(outputs, {payFees})` / `makeIntent(inputs, outputs, {...})` | build a transfer / unbalanced intent |
| `balanceUnsealedTransaction(tx)` / `balanceSealedTransaction(tx)` | balance + pay fees for a contract-built tx |
| `signData(data, {encoding, keyType})` | sign arbitrary data with the unshielded key |
| `submitTransaction(tx)` | broadcast a sealed tx — *the wallet acts as relayer* |
| `getProvingProvider(keyMaterialProvider)` | delegate ZK proving to the wallet |
| `getConfiguration()` / `getConnectionStatus()` / `hintUsage(methodNames)` | config; status (+networkId); pre-request hints |

:::warning[Feature-detect — coverage varies]

The connector is a contract, but not every wallet implements
every method. **Lace does not implement `getProvingProvider()` or `signData()`** (see [Lace](#lace)).
Always check, e.g. `typeof api.getProvingProvider === "function"`, before calling.

:::

:::warning[The #1 connect gotcha: call connect() synchronously inside the click handler]

Lace opens a real
authorization pop-up, and the browser silently blocks it if transient user activation was lost (you
`await`ed something, or used `setTimeout`/an rxjs `interval` first). For page-load auto-reconnect there
is no gesture — poll `window.midnight` until injected (extensions inject slightly after
`DOMContentLoaded`; users may need to **refresh after installing**), then connect.

:::

### 5.3 Where ZK proofs come from

A Midnight tx needs a ZK proof; *who generates it* is the main axis between wallets (and maps onto the
custody models):

- **Local proof server (Lace).** Lace requires a **local proof server** (Settings » Midnight » Local,
  `http://localhost:6300`). Witness data stays on the user's machine — but they must run the server, and
  Lace does **not** expose `getProvingProvider()`, so a dApp can't delegate proving to it.
- **In-browser WASM (1AM).** 1AM compiles the prover to WebAssembly (Halo2, ~2.9 MB) and proves *in the
  tab* — no separate process, sub-second after a cold-start load. 1AM **does** implement
  `getProvingProvider()` and offers a hosted **Proof Station**.
- **Delegated (`getProvingProvider`).** The v4 connector abstracts proving behind it; the old
  `Configuration.proverServerUri` is **deprecated**. Obtain a proving provider from the wallet when
  available, and fall back to a configured proof server for Lace.

### 5.4 Fees, DUST, and "gas sponsorship"

Fees are paid in DUST, generated by holding NIGHT and regenerating over time. `getDustBalance()` returns
`{ balance, cap }`, and `submitTransaction` "uses the wallet as a relayer" with a `payFees` option — so a
wallet *can* sponsor fees, but **sponsorship is a capability the connector allows, not a guarantee any
wallet ships**. On a fresh wallet, DUST takes time to generate — **≈12 h on Lace mainnet/testnet, ≈5 min
on a local network** — and overspending surfaces as `BalanceCheckOverspend` (138)
([Troubleshooting](#10-troubleshooting-and-error-reference)).

### 5.5 Integrate: the portable code

Works for any standard-connector wallet (Lace, 1AM); the [CLI connector](#8-develop-and-test-without-a-wallet)
is a no-extension dev fallback.

```ts
import { setNetworkId } from "@midnight-ntwrk/midnight-js-network-id";
import type { InitialAPI, ConnectedAPI } from "@midnight-ntwrk/dapp-connector-api";

// 1. Discover (legacy key first, then v4 rdns scan)
const wallets = Object.values(window.midnight ?? {})
  .filter((w): w is InitialAPI => !!w?.name && !!w?.apiVersion);

// 2. Connect — MUST be synchronous in the click handler (pop-up blocking)
//    networkId: only 'mainnet' is standard; others are wallet-defined (CLI uses 'Preview' etc.)
async function connect(w: InitialAPI, networkId = "preprod"): Promise<ConnectedAPI> {
  const api = await w.connect(networkId);             // user authorizes here
  const status = await api.getConnectionStatus();
  if (status.status !== "connected") throw new Error("wallet disconnected");
  setNetworkId(status.networkId);                     // align the dApp to the wallet's network
  return api;
}
// To submit: build the tx (api.makeTransfer / your contract), prove (feature-detect
// api.getProvingProvider, else local proof server), then api.submitTransaction(tx).
```

**Where wallets diverge — handle these:**

- **Proving** — feature-detect `getProvingProvider`; 1AM has it, **Lace doesn't** (fall back to the local
  proof server; don't hard-code the deprecated `proverServerUri`).
- **`signData`** — not on Lace; feature-detect before signature-based auth.
- **Pop-up** — Lace opens an auth pop-up (keep `connect()` synchronous); the reconnect path has no
  gesture, so poll `window.midnight` then connect.
- **Network** — always reconcile the dApp to `getConnectionStatus().networkId`.
- **DUST** — read `getDustBalance()`, handle `{ balance: 0n }` and the `cap`; don't assume sponsored fees.
- **Non-standard tiles** — Ctrl/Gero may not inject a conformant connector; feature-detect and degrade to
  Lace/1AM.

### 5.6 React (the production pattern)

The Edda Labs **midnight-starter-template** (demo at `counter.nebula.builders`) wires Lace + 1AM through a
copy-pasteable wallet-widget — use it rather than re-deriving:

```tsx
import { useWallet } from "@/modules/midnight/wallet-widget/hooks/useWallet";

function ConnectButton() {
  const { connectWallet, disconnect, status, dustBalance } = useWallet();
  if (status?.status === "connected")
    return <button onClick={() => disconnect()}>Connected · DUST {String(dustBalance?.balance)}</button>;
  return <button onClick={() => connectWallet("mnLace", "preprod")}>Connect Lace</button>;
}
```

### Lace

> **Footprint:** Self-custody × **Browser** (Midnight); mobile/hardware are Cardano-side only.

> **Best for:** standard dApps whose users can run a local proof server.

> **Footprint:** Self-custody × **Browser** + **Mobile** (beta); drivable from code via the connector.

> **Best for:** new Midnight-first dApps; users wanting shielded-by-default + mobile.
> **Status:** live; implements `getProvingProvider` (in-browser proving). **Open source:** not published.

- **Overview:** purpose-built for Midnight. Extensions for **Chrome + Firefox** live; **iOS/Android in

- **Not documented:** no Cardano / multi-chain support; hardware support and open-source status are not

### Kuira

> **Footprint:** Self-custody × **Mobile** (Android) + an embeddable **SDK** dev path; on-device proving.
> **Verdict:** the concrete entry in the long-empty mobile cell — a wallet apps carry *inside* themselves.
> **Best for:** mobile-first apps that want a built-in wallet (no separate install, no seed phrase).
> **Status:** alpha (`v0.1.0-alpha02`). **Open source:** partial (source jars on Maven Central; repo private).

- **Overview:** a native **Android** wallet **and** an embeddable SDK — drop the wallet *inside* an
  Android app so it can hold and move funds on Midnight by itself, no external wallet required. Built by
  **Kuira Labs**; distributed on Maven Central (`io.github.kuiralabs:dapp-ui:0.1.0-alpha02`); **Apache-2.0**
  but the **source repo is private** (only source jars ship). Public developer docs.
  ([Kuira SDK docs](https://kuiralabs.github.io/kuira-sdk-android/))
- **Supports:** shielded **and** unshielded transactions; **on-device ZK proving** (in seconds, no proof
  server to run — unlike Lace's local server or 1AM's in-browser WASM); an **embedded self-custody wallet**
  in the app's own process; DUST regeneration + indexer sync; example apps (a Kuira Starter counter and
  **BBoard**, a shared on-chain bulletin board).
- **Identity — the "Sigil":** in plain terms, your identity is a **passkey + fingerprint** instead of a
  seed phrase — Kuira calls it a *Sigil*. Under the hood it's a **passkey-derived DID (decentralized
  identifier)** minted by a single biometric, so onboarding is seedless and you recover on any device that
  shares the Google account. The `SigilIdentityProvider` is a swappable seam:
  **[Midnight Passport](#midnight-passport-announced)** is named as a planned future identity backend — so
  the "announced" Passport row and this "emerging mobile" cell may converge (Passport as the identity,
  Kuira as where it lives on the phone).
- **Doesn't (yet):** **Android-only today** (iOS planned; the vendor says iOS + React Native later in
  2026), so the SDK path is platform-locked for now; the browser **DApp Connector is not documented** (a
  `connector` module exists, but conformance with the `window.midnight` standard isn't publicly stated);
  alpha, with documented limitations closing in future releases.

> **Status:** announced; publicly previewed (~2026–2027). **Built by:** IOG.

- **What it is:** an **announced IOG initiative**, introduced by Charles Hoskinson in **April 2026** —

  with no seed phrase**. Based on what has been shared publicly, it features: **per-device,

  publicly, so it's placed in this row provisionally pending complete specifications.
- **Status:** full technical specifications have not yet been published, and as an announced product it is
  **not yet available for integration** — included here as a signal of where Midnight's consumer
  onboarding is heading.

## 8. Develop and test without a wallet

The **Self-custody × CLI** cell in practice. The **wallet CLI** exposes the **same `ConnectedAPI`** over a
local WebSocket, so dApp code paths run with zero changes — then users connect with any browser wallet.
The CLI is **self-custody** (keys in `~/.midnight`, BIP-39/44 — Lace-compatible) and **network-agnostic**
(one seed → `undeployed`, `preprod`, `preview`).

| Group | Commands |
|---|---|
| Wallet | `wallet generate/list/use/info/remove`, `info`, `address`, `genesis-address` |
| Transactions | `balance`, `transfer`, `airdrop`, `inspect-cost` |
| DUST | `dust register`, `dust status` |
| Local network | `localnet up/status/stop/down/clean` |
| Config / connector | `config get/set/unset`, `cache clear`, `serve` |

`midnight localnet up` orchestrates the full stack in Docker — **node (`9944`), indexer (`8088`), proof
server (`6300`)** — so you can deploy and transact with no faucet:

```bash
midnight localnet up                          # node + indexer + proof server (Docker)
midnight wallet generate alice                # self-custody key in ~/.midnight
midnight config set network undeployed
midnight airdrop 1000                          # fund the active wallet (undeployed-only)
midnight airdrop 1000 mn_addr_undeployed1...   # …or fund ANY address directly (v0.4.1 — no seed import)
midnight dust register                         # REQUIRED before transfers (else error 138)
midnight balance
midnight transfer mn_addr_undeployed1... 100   # positional: <recipient> <amount>

midnight serve --network preview               # headless DApp Connector on ws://localhost:9932
midnight serve --network preview --approve-all # dev only: auto-approve every request
```

:::warning[#1 beginner gotcha]

`dust register` **before** transferring, or it fails with
`BalanceCheckOverspend` (138). DUST regenerates ≈5 min locally
([§5.4](#54-fees-dust-and-gas-sponsorship), [Troubleshooting](#10-troubleshooting-and-error-reference)).

:::

```ts
// A dApp connects to the CLI connector exactly like a browser wallet:
import { createWalletClient } from "midnight-wallet-connector";
const wallet = await createWalletClient({ url: "ws://localhost:9932", networkId: "Preview" });
const balances = await wallet.getUnshieldedBalances(); // same ConnectedAPI surface
```

Build/test against `midnight serve` (fast, scriptable, no pop-ups), then deploy and let users connect with
Lace/1AM — zero code changes.

`midnight airdrop` can now **fund any address directly** — paste the
recipient's address, no seed import — which removes a real annoyance when funding browser wallets on a

## 10. Troubleshooting and error reference

Symptom → cause → fix. First the **ledger / connector** errors; then the **CLI JSON error contract**
(`--json` → `{error, code, message, exitCode}`).

| Symptom / error | Likely cause | Fix |
|---|---|---|
| `BalanceCheckOverspend` (**138**) | Not enough DUST for fees | Wait for DUST (≈12 h fresh Lace; ≈5 min local) or designate more NIGHT; read `getDustBalance()` first |
| Authorize pop-up never appears / connect hangs | `connect()` not called synchronously on click | Call `connect()` directly in the click handler; don't `await`/`setTimeout` before it |
| `window.midnight` is `undefined` | Extension not installed/enabled, or injected after load | Prompt to install, then **refresh**; for auto-reconnect, poll before connecting |
| `getProvingProvider is not a function` (Lace) | Lace doesn't implement it | Feature-detect; use a local proof server — [lace#2224](https://github.com/input-output-hk/lace/issues) |
| `signData` "Method not implemented" (Lace) | Unimplemented on Lace | Avoid signature-based auth on Lace — [lace#2165](https://github.com/input-output-hk/lace/issues) |
| Proof generation fails / prover unreachable | Local proof server down, or Brave Shields blocking it | Start it (`localhost:6300`); disable Brave Shields for the origin |
| Network mismatch | dApp and wallet on different networks | Reconcile to `getConnectionStatus().networkId` via `setNetworkId(...)` |
| 1AM first proof very slow | ~2.9 MB WASM cold-start | Pre-warm / show a "loading prover" state |
| Deploy fails before it starts | Wallet not fully synced | Wait for wallet sync to complete before deploying |
| CLI `DUST_REQUIRED` (exit 5) | No DUST | `midnight dust register`, retry once DUST generates |
| CLI `INSUFFICIENT_BALANCE` (exit 5) | Not enough NIGHT | Fund the wallet (faucet on testnets; `airdrop` only on `undeployed`) |
| CLI `NETWORK_ERROR` (exit 4) | Connection refused / timeout / DNS | Check node/indexer endpoints |
| CLI `PROOF_TIMEOUT` (exit 6) | ZK proof timed out | Ensure the proof server is healthy; retry; reduce circuit complexity |
| CLI `STALE_UTXO` (exit 6) | UTXOs consumed by another tx | `midnight cache clear` and rebuild |
| CLI `TX_REJECTED` (exit 6) | Node rejected the tx (TTL / validity) | Inspect the message; rebuild with current state |

CLI codes are a documented contract (see the [CLI's JSON-output reference](https://www.npmjs.com/package/midnight-wallet-cli));
for canonical Midnight error/status codes, use the `midnight-status-codes` reference.

## 11. Security checklist

The connector hands your page data from an untrusted extension — treat it accordingly:

- **Sanitize the wallet `name`**; render the `icon` via `<img src>` — never `innerHTML` (XSS).
- **Validate `networkId`** — reconcile to `getConnectionStatus().networkId` and refuse mismatches.
- **Don't trust injected data as authority** — `rdns`/`name` are for display/selection, not security
  decisions; match the expected `rdns` and tell users to install only from official stores.
- **Keep witness data local** — prefer in-browser proving (1AM) or a local proof server (Lace); if you
  ever use a hosted prover, say so.
- **Confirm fee sponsorship** — not all wallets sponsor fees; verify on your target network before
  promising a "no DUST needed" experience.
- **Guard agent (MCP) access hard** — scope tools, cap spend per action and per day, allowlist
  destinations, require human confirmation for anything irreversible. An agent with unbounded signing
  authority is the highest-risk cell on the grid.

## 14. Glossary

Wallet-relevant terms.

- **Custody model** — *who controls the keys*; the matrix rows (self-custody, custodial, AA, MPC). Maps to
  Full Sovereignty → Delegated → Smart Custody.
- **Self-custody / Custodial** — keys on the user's device (no third party can sign) / a provider holds
  and signs.
- **Account Abstraction (AA)** — the account is on-chain code with programmable auth. **Not native to
  Midnight as of June 2026.**
- **MPC / threshold (TSS)** — the key is split into shares; an *m*-of-*n* quorum signs without
  reconstructing a full key. Basis of institutional custody (Fireblocks for NIGHT).
- **Interface** — *how the wallet is reached*; the matrix columns (browser, mobile, CLI, MCP).
- **MCP (Model Context Protocol)** — a standard by which AI agents call tools — wallet tools (move funds)
  or developer tools (help you build).
- **Coverage footprint** — the *set of cells* a wallet occupies; reading it shows who can use it and how
  far it reaches.
- **Midnight Passport** — *announced* first-party seedless, QR-onboarded consumer wallet (browser +
  mobile) from IOG; per-device hardware-resident keys with trusted-helper recovery. Exact scheme (MPC vs.
  secure-enclave) **not yet publicly detailed** ([§6](#midnight-passport-announced)).
- **Sigil** — [Kuira](#kuira)'s identity primitive: a **passkey-derived DID** minted by one biometric (no
  seed phrase, seedless onboarding). Exposed in the SDK as a swappable `SigilIdentityProvider`, with
  [Midnight Passport](#midnight-passport-announced) named as a planned future backend.
- **On-device proving** — generating a transaction's ZK proof **on the user's phone** (Kuira, Android) in
  seconds, with no local proof server to run and nothing leaving the device — the mobile counterpart to
  1AM's in-browser WASM proving.
- **NIGHT / DUST** — native value token (public; generates DUST) / shielded, non-transferable, decaying
  fee resource with a `balance` and a `cap`.
- **Shielded vs unshielded** — Midnight's dual ledger: private vs public state. 1AM/urble default to
  shielded; Lace is opt-in.
- **Selective disclosure / viewing key** — reveal specific facts without exposing full data / a read-only
  key for auditors.
- **DApp Connector / CAIP-372** — the standard by which a wallet injects an API under `window.midnight`.
- **rdns** — reverse-DNS wallet id used for discovery (`mnLace`, `'1am'`).
- **Intent** — an (often unbalanced) set of inputs/outputs; the building block for transfers/swaps.
- **Sealed vs unsealed tx** — ready-to-submit (proofs + binding) vs pre-binding.
- **Relayer** — submits a tx and pays its fees on the user's behalf (the basis of "gas sponsorship").
- **Proving provider** — the abstraction (`getProvingProvider`) for delegating ZK proving to the wallet.
- **Bech32m** — the address encoding the connector returns.
- **Local proof server** — the native binary (`localhost:6300`) that generates ZK proofs for wallets that
  don't prove in-browser.
- **Wallet-IaaS / Smart Custody** — Midnight infrastructure for delegated/embedded wallet provisioning
  (a.k.a. **Wallet-as-a-Service, WaaS**); the route for social/passkey onboarding.
- **DID (decentralized identifier)** — a self-owned identifier (e.g. `did:key:…`) not issued by a central
  authority; in Kuira a DID is derived from a passkey to form the [Sigil](#14-glossary).
